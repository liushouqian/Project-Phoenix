# Day 7

## Sprint
Sprint 1

## Sprint Task
Task 5 - Conversation Management

## Today's Goal
Add in-memory conversation history support for /chat API.

## Finished
- Added session_id to ChatRequest
- Updated /chat route to pass session_id and message into generate_reply
- Implemented in-memory conversation storeage with conversation_store
- Added get_history(session_id) and append_message(session_id, role, content)
- Injected history into LLM request messages
- Verified multi-turn conversation works with the same session id
- Confirmed the assistant can remember user name and city across turns

## Problems
- At first, I hit a NoneType-related issue when trying to inject conversation history into the model input.
- I also ran into several model access issues while testing free endpoints, including 404 and 429.
- During testing in Swagger, I initially got confused by request body changes and response behavior, which made debugging slower.

## How I Solved Them
- I changed the history getter to return an empty list by default instead of None.
- I switched to a paid DeepSeek-compatible API setup to make the chat service stable.
- I added debug logs for session_id, message, and history before/after, then verified the full conversation flow step by step.

## What I Learned
- route is responsible for HTTP request/response handling.
- service is responsible for business logic such as model calling and history assembly.
- Conversation memory can be implemented by storing messages under a session_id.
- Debugging AI applications often means checking data flow, not just checking the model ouput.
- Multi-turn chat is essentially "message history + model call + history persistence".
