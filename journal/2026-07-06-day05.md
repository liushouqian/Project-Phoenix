## Finished
- Connected the `/chat` endpoint to a real LLM provider
- Added OpenAI-compatible client configuration
- Loaded API settings from `.env`
- Verified `/health` endpoint
- Verified `/chat` endpoint with Swagger UI
- Fixed Git issues related to `.venv` and `__pycache__`
- Added `.env.example` for environment setup

## Problems
- At first, the `/chat` endpoint returned a generic self-introduction no matter what message I sent.
- I suspected the backend code might still be using mock logic, but after checking the route and service files, the code path was correct.
- The real issue was the model configuration. I used `openrouter/free`, and its behavior was unreliable for this project.
- After switching to a specific free model, the responses became normal and matched the user input.
- I also ran into a GitHub push protection error because a secret-like API key string was included in `.env.example`. I fixed it by replacing it with a placeholder value.

## What I Learned
- A successful API response does not mean the AI feature is correct.
- In AI engineering, debugging includes not only Python code and API routes, but also model configuration and provider behavior.
- It is safer to use a specific model name instead of a generic provider alias.
- `.env.example` should only contain placeholder values and must never include real secrets.

## Reflection
Today was the first day that the project felt like a real AI backend instead of a normal backend exercise.
I learned that AI engineering problems can come from many layers: code, environment, model provider, and even repository security rules.
This made me realize that “the API works” is not enough — I also need to verify whether the model behavior is actually correct.
