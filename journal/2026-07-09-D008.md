## Today's Goal

- system prompt
- AI assistant role

It was mainly a refinement and verification task based on the existing chat architecture.

Before Task 6, the project allready had:

- FastAPI `/chat` endpoint

- primary / fallback model logic

Task 6 focused on making the AI behavior more explicit and more controllable.

## Completed
- Created / refined `prompts.py`
- Added a structured `SYSTEM_PROMPT`
- Defined Phoenix role and response rules
- Added `history before` debug logging
- Added `history after` logging in both primary success and fallback success branches
- Verified prompt behavior with dedicated test sessions

## Prompt Design

The final prompt defines Phoenix as:

- an AI engineering assistant
- focused on helping a frontend engineer transition into an AI engineer
- practical, concise, and technically accurate

Rules added in prompt:

1. Focus on engineering growth, practical learning, and project building
2. Prefer clear next steps over vague motivation
3. Do not invent user facts that are not provided in the current conversation history
4. If uncertain, say so clearly

## Verification

### Test 1: Identity

Question:
你是谁？

Result:
Phoenix correctly introduced itself as an AI engineering assistant for frontend-to-AI transition.

### Test 2: Engineering guidance style

Question:
我是一个有10年前端经验的工程师，现在想转AI工程师，你建议我下一步学习什么？

Result:
Phoenix returned a practical engineering-oriented answer instead of generic motivational advice.

### Test 3: Anti-hallucination boundary

Question:
你知道我的生日吗？

Result:
Phoenix answered that it did not know, instead of inventing a fake answer.

## What I Learned

### 1. Prompt is part of product behavior

Prompt is not just text for the model.
It defines how the AI assistant should behave inside the product.

### 2. A chat application needs behavior boundaries

The assistant should not only answer questions.
It should also follow product rules such as:

- role consistency
- engineering focus
- anti-hallucination boundary

### 3. Debug logs are part of AI engineering

By logging: 

- incoming session_id
- incoming message
- history before
- history after

I can better understand whether the issue comes from:
- prompt 
- memory
- LLM response
- fallback logic
